The one idea that unlocks it

onSite lets an optional worker ride along on an appointment and skip the rule checks. Almost every finding is a consequence of that one design choice — or its absence.

Refuse ≠ crash

Both products often stop a bad booking. The question is how: a clean 400 you can act on, or a 500 server crash. Two of the divergences are exactly this.

Read path ≠ write path

Showing a slot and booking it run different code. Several gaps are one path enforcing a rule the other skips — the write-path double-book is the big one.

The three engines, in one picture each

Same request. Three fundamentally different machines.

InField
Optimize the route
Sends a technician to the customer. A job lists three ID lists — required / preferred / avoid — and the O2 engine assigns exactly one resource per route. No lead worker, no per-worker flags, no seat to ride along in.
Scheduler (old)
Attach & check the lead
Customer comes to a branch. Attaches workers to an appointment, marks one primary, flags each required or not. A not-required worker rides along and skips the rule checks.
Unified (new)
Same idea, rewritten
Meant to replace Scheduler. Same attach-and-flag model, separate codebase. Agrees with Scheduler almost everywhere — the divergences are where the rewrite drifted.

The whole comparison turns on one line: InField has no optional-resource seat. Scheduler and Unified both do — so their bugs live in that seat, and InField is immune to them by construction.

The trap everyone falls into

InField's "crew member" looks like onSite's optional worker.
It is the exact opposite.

onSite · optional resource
An exception to the rules
Attached to the appointment as a second bookable seat. It skips skill, territory, location-role, and availability checks. This one seat is the source of behaviors 1a–1.5.
InField · crew member
More input to the rules
Folded into one crew resource: its skills add to the crew, its unavailability clamps the crew, crew-size is a hard constraint. It tightens the checks — it never escapes them.

onSite's optional worker is an exception to the rules. InField's crew member is more input to the same rules. That single inversion is why behaviors 1a, 1b, 1c, 1.4 and 1.5 have nothing to test in InField.

The headline finding of Act A

InField can't double-book. Not by a check — by its shape.

One resource is one route: an ordered list of stops, each starting after the previous stop's departure plus travel. Two jobs sharing wall-clock time on one resource cannot be expressed. There's no flag to turn it off.

Stops chain in time. There is no vertical axis to stack a second overlapping job on — overlap is unrepresentable, not merely rejected.

The one caveat we owe you

The invariant is per-route. Genuinely concurrent overlapping shifts for the same resource are chained by separate machinery we didn't fully exhaust. Flagged honestly, not swept.

▲Two paths bypass the optimizer

Act A · what the InField team should walk away with

InField isn't a looser onSite. It's a stricter, differently-shaped one.

Don't expect the loophole

The onSite "hidden but still bookable" excluded-worker gap does not exist here. Excluded genuinely blocks. Plan around stricter, not laxer.

Half the onSite bugs are N/A

The primary/required-flag family (3, 4, 4z, 5) probes inputs InField can't express. No fix carries over — but no equivalent risk exists either.

Immunity has a footnote

Double-book is impossible on optimizer-backed paths. The two engineless write paths are the exception worth a decision.

▲ Decision for the InField team · 264
Is the engineless drag-and-drop fallback intended — and if so, should it still enforce a basic overlap/shift guard before writing?
Say this to open it
"Everywhere the optimizer runs, you literally cannot double-book. But two paths skip the optimizer and write raw coordinates. That's the one place InField behaves like Unified's write-path bug — worth a conscious call, not an accident."

Divergence 1 & 2 · the easy ones

Both refuse the bad booking. Only the manner differs.

Neither of these is a wrong outcome — both products correctly stop the booking. Scheduler stops it with a message a client can act on; Unified stops it by falling over. Same underlying NPE family both times.

Case A · demand an optional-only worker

Customer demands worker B, but B is present only as an optional resource — the demand can't be met. serviceTerritoryMembers null-pointer.

Case B · omit the required flag

Send a worker without isRequiredResource. Unified reads the missing flag and unboxes a null Boolean. Same crash family.

✕ Decision · Unified engineering · agreed action, not an open question
Null-guard both inputs and return a clean 400, matching Scheduler.
  • Guard serviceTerritoryMembers before .values() (the demand-a-non-required-worker path and the required-excluded write both hit this).
  • Null-guard isRequiredResource unboxing; return the same "specify a valid value" 400.
Say this
"These two aren't behavior disagreements — you already refuse correctly. You just refuse by crashing. Turn the 500 into the 400 Scheduler already returns and both are closed."

Divergence 3 · why it's a contract difference, not a config gap

Unified computes occupancy correctly — on the read. The write just doesn't ask.

Divergence 3 · the two calls to make

Two open questions for Unified

▲ Question A · correctness
Should the write check every required worker's availability — not just the primary?
  • Today only the primary is guarded (containsKey); a required helper with zero slots silently drops out and gets booked over.
  • Fix the intersection to fail when any required worker has no slot, mirroring the read path that already gets this right.
◐ Question B · product
Should Unified expose an overbooking control like Scheduler's two knobs?
  • So double-booking is a deliberate opt-in, not automatic behavior.
  • Keep the multi-resource enable-gate (WorkforceSchdMulResSchdPref) separate — that's a different lever, not an overbooking control.

✓ One myth to kill in the room

The earlier "CalendarEvents rule" theory was wrong — that rule was removed in 264 and even on 262 gated only Salesforce Event records, never ServiceAppointment occupancy. No re-test needed. The root cause is the write-path contract, verified in the debugger on the live build (commit 90cdfeb7, both org URLs resolve to the same box).

Say this to close Act B
"Unified already knows B is busy — its read path says so. The write path just never asks. That's one bug and one product question, not a migration blocker across the board."

The through-line

Decisions 2, 3 and 4 are all the same question in three costumes: which path enforces which rule, and on purpose? Read vs write, required vs optional, engine vs engineless.

One line to leave the room on

These engines mostly agree. The disagreements are a short, owned list.

InField is stricter by construction. Scheduler and Unified are interchangeable on 11 of 13 behaviors. What's left is four decisions — two are already agreed fixes, two are genuine product calls. Nothing here is a mystery; every claim traces to a live test or a file and line.
