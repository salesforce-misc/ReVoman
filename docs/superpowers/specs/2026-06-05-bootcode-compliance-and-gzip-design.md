# Vendored bootcode.js: compliance scrubber + gzip-at-rest

**Date:** 2026-06-05
**Status:** Design approved, pending implementation plan
**Scope:** `build.gradle.kts` (`generatePmSandbox` task), `SandboxResources.kt`, `FileUtils.kt`, committed `postman-sandbox` resources, sandbox tests

## Problem

ReVoman vendors Postman's real sandbox bootcode as a committed classpath resource
(`src/main/resources/postman-sandbox/bootcode.js`, 2.2 MB minified), generated by the
`generatePmSandbox` Gradle task from `postman-sandbox@6.7.0` (+ `uvm` bridge-client). It is
loaded at runtime via `SandboxResources` (classloader-safe `readFileToString` â†’
ClasspathResolver) and evaled in a GraalJS `Context` by `SandboxBridge`.

Two recurring problems with this vendored blob:

1. **Compliance recurrence.** The Salesforce PII/Gov-Cloud scanner (PACE/CRST) does a *naive
   substring match* for forbidden gov-cloud tokens. It flagged `ic.gov` twice â€” both false
   positives inside legit public-suffix-list entries bundled by `tldts` (`vic.gov.au`,
   `ic.gov.pl`). The 2 MB bundle also embeds lodash, chai, moment, cheerio, uuid,
   postman-collection, crypto-js, csv-parse â€” any of which can smuggle a new scanner-forbidden
   token on the next `postman-sandbox` upgrade. The current fix (commit `04a9420`) escaped
   `ic.gov` â†’ `ic\x2egov` as a one-off; this needs to become systematic.

2. **Bloat.** A 2.2 MB minified one-liner is committed to git and re-churns wholesale on every
   upgrade (no useful diffs). It deflates to ~717 KB inside the jar, so the *jar* cost is modest,
   but the *git blob* is 3Ã— larger than necessary.

These two motivations pull slightly differently â€” compliance wants the bytes
transformed/scrubbed, bloat wants them smaller â€” so the design serves both with two composable
layers.

## Verified facts (pre-design investigation)

- **Packaging:** `revoman-0.9.9.jar` contains `postman-sandbox/{bootcode.js, bridge-client.js,
  pm-sandbox-version.txt}`. bootcode is 2,196,743 bytes raw, deflated to 717,405 bytes in the jar
  (67 %). The 0.9.9 jar is compliance-clean (0 `ic.gov`).
- **Runtime works for consumers.** POM declares `org.graalvm.js:js-language` at `scope=runtime`;
  full transitive closure resolves (polyglot, truffle-api, regex, sdk/\*, shadowed/icu4j+xz) at
  25.0.3. In **core** (bazel, the real consumer) graal 25.0.3 is explicitly declared in
  `third_party/dependencies/org_graalvm.bzl` + pinned catalog, and the revoman catalog entry
  lists `@org_graalvm_js_js_language` in its `deps`, fanning the whole engine out to all ~24
  consumers. Core independently uses `polyglot.Context` in `p13n-impl`.
  - **Caveat:** core is pinned at `0.9.7`; it needs a version bump to consume any of this.
- **Loader is classloader-safe.** `SandboxResources` uses `readFileToString` â†’
  `bufferFile` â†’ `resolveClasspath` (thread-context classloader), the documented fix for the
  okio `FileSystem.RESOURCES` classloader gotcha. Preserving this path is a hard requirement.
- **Known cosmetic finding (out of scope here):** `SandboxBridge` does NOT set
  `engine.WarnInterpreterOnly=false` (the old `PostmanSDK.JSEvaluator` does), so it logs a
  multi-line Truffle interpreter-mode warning on every boot on stock (non-JVMCI) JDKs. Tracked
  separately â€” not part of this change.

## Goals

- Eliminate compliance-scan recurrence at the source (any forbidden token, not just `ic.gov`).
- Cut the committed git blob ~3Ã— (2.2 MB â†’ 717 KB).
- Zero change to downstream behavior: `SandboxBridge` and consumers see an identical bootcode
  `String`.
- Preserve the classloader-safe resource resolution.

## Non-goals (YAGNI)

- Slimming/tree-shaking the bundle (Postman owns the bundle; fragile, uncertain payoff).
- Fetch-at-build / un-vendoring (breaks the JVM-first promise; bazel-core can't run npm mid-build).
- Fixing interpreter-mode perf or the boot warning (separate concern).

## Design

Two independent, composable layers. **Scrub runs before gzip** so the clean bytes are what get
compressed.

### Layer A â€” Compliance scrubber (build-time, in `generatePmSandbox`)

- Replace the inline one-off `ic.gov` `node -e` with a scrub step driven by an explicit,
  commented `FORBIDDEN_TOKENS` array (initially `["ic.gov"]`) â€” the single place to add the next
  scanner trip-word.
- For each token, `\xNN`-escape one character (e.g. the `.`) inside JS string literals: the JS
  engine decodes `\x2e` â†’ `.`, so the runtime value is byte-identical, but the literal token bytes
  no longer exist in the file. (Note: `\xNN` is valid in JS string literals but illegal in JSON â€”
  the PSL data is a JS array literal, confirmed by bare unescaped quotes and `node --check`.)
- Apply to **both** `bootcode.js` and `bridge-client.js` (cheap, safe).
- This is the **primary** compliance defense â€” it guarantees clean source-of-truth bytes
  regardless of packaging.

### Layer B â€” Gzip at rest (build-time write + runtime inflate)

- `generatePmSandbox` writes `bootcode.js.gz` (gzip -9, ~717 KB) instead of raw `bootcode.js`,
  and deletes the raw file.
- `bridge-client.js` (3 KB) stays raw â€” gzipping it is pointless.
- `SandboxResources.bootcode` reads the `.gz` resource and inflates lazily (cached for JVM
  lifetime).
- **Bloat win:** committed blob 2.2 MB â†’ 717 KB. Jar weight ~unchanged (already deflated), but no
  double-store.
- **Compliance bonus:** gzip'd bytes are opaque to naive substring scanners â€” defense-in-depth
  behind A. (A still matters for the case where someone reverts to raw.)
- **Runtime cost: negligible.** Measured inflate+UTF-8 decode of the 2 MB bootcode â‰ˆ **7.7 ms**,
  once per JVM (`by lazy`). GraalJS boot/eval of the same bootcode is hundreds of ms to seconds,
  so this is rounding error; likely a wash vs. reading the larger raw resource off the classpath.

## Components & file-level changes

### 1. `build.gradle.kts` â€” `generatePmSandbox` task (only build-time change)

Pipeline (replacing the current bootcode-write + inline ic.gov escape lines):

```
generate bootcode.js (raw)                         # existing node -e
generate bridge-client.js                          # existing node -e
â”€â”€ scrub (A) â”€â”€ node: escape each FORBIDDEN_TOKENS entry in bootcode.js + bridge-client.js
â”€â”€ gzip (B) â”€â”€ gzip -9 bootcode.js â†’ bootcode.js.gz ; rm bootcode.js
write pm-sandbox-version.txt                        # existing node -e
```

- `FORBIDDEN_TOKENS` is a commented array at the top of the task (initially `["ic.gov"]`).
- Scrub strictly before gzip.

### 2. `FileUtils.kt` â€” new classloader-safe gzip reader

```kotlin
fun readGzippedFileToString(filePath: String): String =
    bufferFile(filePath).gzip().buffer().readUtf8()
```

- Reuses the existing `bufferFile` (â†’ `resolveClasspath`, thread-context classloader), so the
  classloader-gotcha fix is preserved with no new code path.
- okio's `GzipSource` (`.gzip()`) validates header/CRC and throws `IOException` on corruption.

### 3. `SandboxResources.kt` â€” runtime loader

```kotlin
val bootcode: String by lazy { readGzippedFileToString("$DIR/bootcode.js.gz") }
val bridgeClient: String by lazy { readFileToString("$DIR/bridge-client.js") }   // unchanged
val version: String by lazy { readFileToString("$DIR/pm-sandbox-version.txt").trim() }
```

- Update the KDoc to note bootcode is gzip-at-rest, inflated once and cached.

### 4. Committed resources

- Delete `src/main/resources/postman-sandbox/bootcode.js` (2.2 MB).
- Add `src/main/resources/postman-sandbox/bootcode.js.gz` (~717 KB, scrubbed-then-gzipped).
- `bridge-client.js`, `pm-sandbox-version.txt` â€” unchanged.

### Data flow (downstream unchanged)

`SandboxResources.bootcode` still returns the same inflated `String` â†’
`SandboxBridge` line 119 `Source.newBuilder("js", bootcode, "bootcode.js")` is untouched. Gzip is
fully encapsulated in the loader. Consumer impact (core/bazel): none beyond the version bump
already required.

## Error handling

- Missing resource â†’ existing `bufferFile` throws `FileNotFoundException("file not found on
  classpath: â€¦")`. No new path.
- Corrupt gzip â†’ okio `GzipSource` throws `IOException` (bad magic/CRC) at first boot â€” fail-fast.
  Acceptable: an immutable build artifact, corruption = packaging bug, not runtime input.
- `by lazy` caching unchanged â€” inflate happens once (~7.7 ms).

## Testing

Subtlety: the compliance scan runs on **file bytes**, not the runtime string. The inflated runtime
string legitimately contains `ic.gov` (inside `vic.gov.au`) â€” that is correct and must not be
asserted away. Gzip (B) alone defeats the scanner (it can't find `ic.gov` in compressed bytes);
scrub (A) defends the revert-to-raw case. Tests therefore pin both invariants without asserting on
the runtime string's substring content.

1. **`SandboxResourcesTest` (update existing):**
   - `loads bootcode bridgeClient and version from classpath` â€” keep `bootcode.length >
     1_000_000`; now also proves gzip inflation end-to-end.
   - `bootcode has no node-vm dependencies` â€” unchanged; passes on inflated content.
   - **New:** `bootcode is gzipped at rest, raw js absent` â€” assert the `bootcode.js.gz` resource
     resolves on the classpath and the raw `bootcode.js` resource does NOT (catches an accidental
     un-gzip regression).

2. **Scrub invariant (build-time logic):** a small unit/fixture test that the scrub transform
   escapes every `FORBIDDEN_TOKENS` entry and that the escaped form decodes to the original (JS
   string-literal decode, NOT `JSON.parse`). May live as a Kotlin test over a fixture string
   mirroring the scrub regex, to avoid a Node dependency in the test JVM.

3. **All 23 existing sandbox tests** (boot, eval, env round-trip) must still pass â€” proves
   inflation feeds GraalJS identically.
