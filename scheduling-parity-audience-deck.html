Set the frame before we start

This is an alignment on behavior — not a bug review.

The goal is a shared understanding of how each engine behaves when it books a worker. Any bugs we point at are a side effect of comparing them, not the reason we're here.

Scheduler vs Unified (onSite, 262)

A limited set of behaviors, mostly around multi-resource bookings — not comprehensive. Across that set they agree; a few disagreements look like bugs.

InField vs onSite

A different machine. Everything built on the Primary resource is N/A — the concept is absent. The optional resource skipping the rules is the root cause of all onSite divergence.

Double-booking

InField allows none — it's impossible by construction. In Unified, overbooking controls simply aren't prioritized yet.

So the through-line isn't "who has bugs." It's where the three engines already behave the same, and where we need to decide they should.

Why behavior alignment matters

A booking that skips a check isn't a technicality. Someone feels it.

We'll say "the booking is wrong" a lot. Here's what "wrong" actually costs — every rule exists to prevent a specific real-world failure.

Wrong person shows up
The customer named a worker they don't want — a past bad visit, a dispute, a safety concern. Skip the check and that exact person arrives at their door.
Nobody shows up
A worker gets booked into two overlapping jobs. They can only be in one place. The second customer waits for someone who is never coming.
The wrong skills show up
A worker who can't actually do the job is dispatched. The visit fails, a second truck-roll is scheduled, and in some trades it's a safety risk.
A demand is silently ignored
The customer required a specific person. The booking goes through without them — and no one is told the demand went unmet.
The app looks broken
Instead of a clear "we can't do that, here's why," the server throws a 500 error. The user sees an outage-like failure with nothing to act on.
The UI lied
A time slot is offered, the customer picks it, and booking then fails. The screen promised something the system won't honor.
The three engines, in one picture each

Same request. Three fundamentally different machines.

InField
Optimize the route
Sends a technician to the customer. A job lists three ID lists — required / preferred / avoid — and the engine assigns exactly one resource per route. No lead worker, no per-worker flags, no seat to ride along in.
Scheduler (old)
Attach & check the lead
Customer comes to a branch. Attaches workers to an appointment, marks one primary, flags each required or not. A not-required worker rides along and skips the rule checks.
Unified (new)
Same idea, rewritten
Meant to replace Scheduler. Same attach-and-flag model, separate codebase. Agrees with Scheduler almost everywhere — the divergences are where the rewrite drifted.

The whole comparison turns on one line: InField has no optional-resource seat. Scheduler and Unified both do — so their rule-skips live in that seat, and InField is immune to them by construction.

The trap everyone falls into

InField's "crew member" looks like onSite's optional worker.
It is the exact opposite.

onSite · optional resource
An exception to the rules
Attached to the appointment as a second bookable seat. It skips skill, territory, location-role, and availability checks — the whole A family lives in this one seat.
InField · crew member
More input to the rules
Folded into one crew resource: its skills add to the crew, its unavailability clamps the crew, crew-size is a hard constraint. It tightens the checks — it never escapes them.

This one seat is the single root cause of every onSite divergence. Because InField has no ride-along seat, the harms in A1–A6 — wrong person, wrong area, wrong skill — simply cannot be set up. There's nothing to skip a check through.

Act A headline · behavior D1

InField can't double-book. Not by a check — by its shape.

One resource is one route: an ordered list of stops, each starting after the previous stop's departure plus travel. Two jobs sharing the same clock time on one worker cannot be expressed. So the "customer waits for someone who never comes" harm is off the table entirely.

Stops chain in time. There is no second axis to stack an overlapping job onto — overlap is unrepresentable, not merely rejected.

The one caveat we owe you

The guarantee is per-route. Genuinely concurrent overlapping shifts for the same resource are chained by separate machinery we didn't fully exhaust — flagged honestly.

▲ Two paths bypass the optimizer

Drag-and-drop has an engineless fallback (writes raw coordinates, no checks); apply-preview replays a stale plan. The one place InField can force-place a worker onto a busy slot.

Act A · what to remember

InField isn't a looser onSite. It's a stricter, differently-shaped one.

Don't expect the loophole

The onSite "hidden but still bookable" excluded-worker gap (A1) does not exist here. Excluded genuinely blocks the booking.

Half the onSite checks are N/A

The primary / required-flag family (B1–B4) probes inputs InField can't express. No fix carries over — and no equivalent harm exists.

Immunity has a footnote

Double-book (D1) is impossible on optimizer-backed paths. The two engineless write paths are the exception worth watching.

For the customer, that means: in InField the wrong person, wrong skill, wrong area, and double-book harms are prevented by the data model itself — not by a rule that a "helper" could slip past.

Divergences A6, B1, D2 · the three crashes

Both stop the bad booking. But a crash and a refusal are not the same answer.

None of these is a wrong outcome — both products correctly stop the booking. The difference is what the customer-facing app receives back, and whether it can tell the user what to do next.

Scheduler · a usable answer
✓ HTTP 400 — a deliberate refusal, named cause. The app can show the user exactly what to fix.
"Specify a valid value for isRequiredResource."
"We couldn't find any resources for your request."
Unified · looks like an outage
✕ HTTP 500 — unhandled server error. No reason, nothing the app can surface to the user.

because this.serviceTerritoryMembers is null

A6 Demand an optional-only worker

Customer requires worker B, but B is only attached as optional. The demand can't be met → null-pointer.

B1 Omit the required flag

Request leaves out isRequiredResource. Unified reads the missing flag as a null Boolean → same crash.

D2 Excluded worker as the lead

Book a do-not-use worker as the required primary. Unified hits the same serviceTerritoryMembers null.

One fix closes all three: null-guard the input and return the clean 400 Scheduler already returns. It's an agreed engineering action, not an open debate.

Divergence D1 · why it's a contract difference, not a config gap

Unified computes availability correctly — on the read. The write just doesn't ask.

✕ Double-books
The booking action persists the requested workers with no occupancy check at all. Same window, same data — opposite result.

The through-line

Every divergence is the same question in different clothes: which path enforces which rule, on purpose? Read vs write, required vs optional, clean refusal vs crash.

One line to leave the room on

The rules mostly hold. Where they don't, we know exactly who feels it.
